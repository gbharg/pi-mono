SMS Consent Handling

AdvancedMD APIs support SMS consent handling for add and update operations and involve adding attributes under the patient and responsible party <contactinfo> node. In order for SMS messages to be sent to the Patient or Responsible Party from AdvancedMD their consent must be set to 1 (Yes). If consent is not given, SMS messages will fail and will not be delivered to the Patient or Responsible Party.

Note: Consent‑related updates are applied only when the MessagingRecipientTextConsents feature is enabled. If MessagingRecipientTextConsents is not enabled, only non‑consent fields are updated, even if consent‑related fields are included in the request.

Attributes

Attribute Description Allowed Values

smsinformationalconsent

Indicates consent for informational SMS

0 (No), 1 (Yes)

smsinformationalconsenttype

Source/type of informational consent

e (electronic), p (paper), v (verbal)

smsmarketingconsent

Indicates consent for marketing SMS

0 (No), 1 (Yes)

smsmarketingconsenttype

Source/type of marketing consent

e (electronic), p (paper), v (verbal)

Note: Consent type values must always be lowercase (e, p, v). Phone type must always be uppercase C.

ProcessRequest Rules (Patient and Responsible Party)

Phone Field Requirements (Add and Update):

  • If adding or updating any consent field, you must include:

    • otherphone (valid phone number)

    • othertype="C"

  • Without both fields, the API will not apply consent changes.

  • If you are only updating phone numbers, consent fields may be omitted.

Add (New Patient or Responsible Party):

Include only the current consent values.

Example:
Copy 

<contactinfo otherphone="OtherPhone" othertype="C"
    smsinformationalconsent="1" smsinformationalconsenttype="p"
    smsmarketingconsent="1" smsmarketingconsenttype="v"/>

Update (Existing Patient or Responsible Party):

If updating any consent, you must pass both old and new values for that consent pair:

  • Old fields prefixed with o_ (e.g., o_smsinformationalconsent, o_smsinformationalconsenttype).

  • New fields without prefix.

If one field from a pair is missing, that specific consent resets to false (only that pair, not both).

Partial Updates Allowed:

You can update only information consent or only marketing consent without affecting the other.

Valid Consent Types:

e – electronic

p – paper

v – verbal

 

Add vs Update Examples

Patient

Add Example:
Copy 
<contactinfo otherphone="OtherPhone" othertype="C"
    smsinformationalconsent="1" smsinformationalconsenttype="p"
    smsmarketingconsent="1" smsmarketingconsenttype="v"/>
Update Example:
Copy 
<contactinfo o_otherphone="OldOtherPhone" otherphone="NewOtherPhone" 
o_othertype="C" othertype="C"
o_smsinformationalconsent="1" smsinformationalconsent="1"
o_smsinformationalconsenttype="p" smsinformationalconsenttype="p"
o_smsmarketingconsent="1" smsmarketingconsent="1"
o_smsmarketingconsenttype="p" smsmarketingconsenttype="p"/>

Responsible Party

Add Example:
Copy 
<contactinfo otherphone="OtherPhone" othertype="C"
smsinformationalconsent="1" smsinformationalconsenttype="p"
smsmarketingconsent="1" smsmarketingconsenttype="v"/>
Update Example:
Copy 
<contactinfo o_otherphone="OldOtherPhone" otherphone="NewOtherPhone"
o_othertype="C" othertype="C"
o_smsinformationalconsent="0" smsinformationalconsent="1"
o_smsinformationalconsenttype="p" smsinformationalconsenttype="v"
o_smsmarketingconsent="1" smsmarketingconsent="1"
o_smsmarketingconsenttype="p" smsmarketingconsenttype="p"/>

 

Behavior Scenarios

Scenario Required Fields

Add consents (Patient or RP)

otherphone, othertype, consent pairs

Update only phone number

otherphone, othertype

Update only informational consent

otherphone, othertype, old & new informational consent fields

Update only marketing consent

otherphone, othertype, old & new marketing consent fields

Update both consents

otherphone, othertype, old & new fields for both informational and marketing consent pairs

Missing one field in a consent pair

That consent resets to false (only that pair)

 

AdvancedMD.API.Demographic

Add Patient Example:

Endpoint: api/demographic/patientsPOST

Request Body

Copy 
{
  ...
  other: "1111111111",
  phonetype: "C",
  ...
  smsinformationalconsent: true,
  smsinformationalconsenttype: "e",
  smsmarketingconsent: true,
  smsmarketingconsenttype: "v",
  ...
}

Note: Cell phone should be provided, otherwise consent won't be saved.

Update Patient Example:

Endpoint: api/demographic/patients/{patientId} PATCH

Request Body

Copy 
{
...
originalpatient: {
  ...
  other: "1111111111",
  phonetype: "C",
  smsinformationalconsent: true,
  smsinformationalconsenttype: "e",
  smsmarketingconsent: true,
  smsmarketingconsenttype: "v",
  ...
}
updatedpatient: {
  ...
  other: "1111111111",
  phonetype: "C",
  smsinformationalconsent: true,
  smsinformationalconsenttype: "e",
  smsmarketingconsent: false,
  smsmarketingconsenttype: null,
  ...
}
...
}
Get Patient Example:

Endpoint: api/demographic/patients/{patientId} GET

Response Body

Copy 
{
 ...
 contact: {
  ...
  smsinformationalconsent: true,
  smsinformationalconsentchangedby: "PPMD_USER",
  smsinformationalconsentedat: "2026-01-09T06:03:08.61",
  smsinformationalconsenttype: "v",
  smsmarketingconsent: false,
  smsmarketingconsentchangedby: "PPMD_USER",
  smsmarketingconsentedat: null,
  smsmarketingconsenttype: null,
  ...
 }
 ...
}

Constraints

  • smsinformationalconsent, smsmarketingconsent – Boolean;

  • smsinformationalconsenttype, smsmarketingconsenttype – possible values: null, “e“, “v“, “p“;

  • smsinformationalconsentedat, smsmarketingconsentedat – DateTime

  • smsinformationalconsentchangedby, smsmarketingconsentchangedby – string
